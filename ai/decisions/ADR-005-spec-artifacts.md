# ADR-005: goat-plan produces structured spec artifacts

**Date:** 2026-03-22
**Status:** Accepted

## Context

goat-review already has a conditional Phase 0 that checks for `requirements-{feature}.md` and validates diffs against its acceptance tests (M2.8). However, no skill was responsible for producing the structured format that Phase 0 expects. The requirements files generated by goat-plan Phase 1 were narrative-only -- they lacked parseable requirement IDs and binary acceptance tests.

Without a structured format, goat-review's spec compliance check had nothing machine-readable to verify against.

## Decision

goat-plan Phase 1 now produces a structured requirements section inside `requirements-{feature}.md` for Standard complexity or higher. The format uses:

- Sequential `REQ-NNN` IDs for each requirement
- A **Description** and binary **Acceptance Test** per requirement
- Per-requirement **Status** tracking (`pending | pass | fail`)
- A **Non-Requirements** section for explicitly excluded scope
- A **Verification Summary** with counts

The template lives at `workflow/templates/requirements-template.md`. The structured section is appended after the existing narrative brief sections -- it does not replace them. Hotfix-complexity features skip it.

## Consequences

- goat-review Phase 0 can now parse `REQ-NNN` entries and check each acceptance test against the diff
- goat-plan and goat-review are connected through a shared artifact format without tight coupling
- The `REQ-NNN` IDs provide stable references for milestone tasks and review findings
- Adding new fields to the template (e.g., priority, category) requires only a template update, not a skill rewrite
